# 統合テスト手順（手動動作確認）

## 概要

このプロジェクトはテストフレームワークを使用しないため、
開発サーバーを起動して手動でエンドツーエンドの動作を確認します。

## 前提

- `apps/server/.dev.vars` に `SESSION_SECRET` が設定済み
- `pnpm install` が完了済み
- 開発サーバーが `http://localhost:5173` で起動済み (`pnpm dev`)

---

## テストシナリオ 1: ユーザー登録フロー

### 1-1. 新規ユーザー登録（正常系）

1. ブラウザで `http://localhost:5173/register` にアクセス
2. 以下の情報を入力:
   - 名前: `テストユーザー`
   - メールアドレス: `test@example.com`
   - パスワード: `password123`
3. 「登録」ボタンをクリック
4. **確認事項**:
   - ホームページ (`/`) にリダイレクトされること
   - ナビゲーションバーに「テストユーザー」が表示されること
   - セッション Cookie (`session`) が Set-Cookie ヘッダーで設定されること（開発者ツール > ネットワーク）

### 1-2. 重複メールアドレスのエラー

1. 再度 `http://localhost:5173/register` にアクセス
2. 同じメールアドレス `test@example.com` で登録
3. **確認事項**:
   - エラーメッセージ「このメールアドレスはすでに登録されています」が表示されること
   - 登録フォームのページに留まること

### 1-3. バリデーションエラー

1. `http://localhost:5173/register` にアクセス
2. 空のフォームで「登録」ボタンをクリック
3. **確認事項**:
   - 各フィールドにバリデーションエラーが表示されること

---

## テストシナリオ 2: ログイン・ログアウトフロー

### 2-1. ログイン（正常系）

前提: シナリオ 1-1 でユーザー登録済み

1. ブラウザで `http://localhost:5173/login` にアクセス
2. 以下を入力:
   - メールアドレス: `test@example.com`
   - パスワード: `password123`
3. 「ログイン」ボタンをクリック
4. **確認事項**:
   - ホームページ (`/`) にリダイレクトされること
   - ナビゲーションバーに「テストユーザー」が表示されること

### 2-2. ログイン失敗（誤パスワード）

1. `http://localhost:5173/login` にアクセス
2. 正しいメールと誤ったパスワードを入力
3. **確認事項**:
   - エラーメッセージ「メールアドレスまたはパスワードが間違っています」が表示されること
   - ログインページに留まること

### 2-3. ログアウト

1. ログイン済みの状態で「ログアウト」ボタンをクリック
2. **確認事項**:
   - ログインページ (`/login`) にリダイレクトされること
   - ナビゲーションバーのユーザー名が消えること
   - セッション Cookie が削除されること（開発者ツール > Application > Cookies）

### 2-4. 未認証でのアクセス制御

1. ログアウト状態で `http://localhost:5173/posts/new` にアクセス
2. **確認事項**:
   - ログインページ (`/login`) にリダイレクトされること

---

## テストシナリオ 3: ブログ記事 CRUD フロー

前提: ログイン済み状態

### 3-1. 記事一覧表示

1. `http://localhost:5173/` または `http://localhost:5173/posts` にアクセス
2. **確認事項**:
   - 初期モックデータの記事一覧が表示されること
   - 各記事にタイトル・著者・作成日が表示されること
   - 「新規記事を書く」ボタンが表示されること（ログイン時）

### 3-2. 記事作成（正常系）

1. `http://localhost:5173/posts/new` にアクセス
2. 以下を入力:
   - タイトル: `テスト記事のタイトル`
   - 本文 (Markdown):
     ```markdown
     # 見出し

     これは **テスト記事** です。

     - リスト項目 1
     - リスト項目 2
     ```
3. 「投稿する」ボタンをクリック
4. **確認事項**:
   - 作成した記事の詳細ページにリダイレクトされること
   - タイトルが正しく表示されること
   - Markdown が HTML にレンダリングされていること（太字・リスト等）
   - 著者名が表示されること

### 3-3. 記事詳細表示

1. 記事一覧から任意の記事をクリック
2. **確認事項**:
   - 記事タイトル・本文・著者・作成日が表示されること
   - 自分の記事の場合、「編集」「削除」ボタンが表示されること
   - 他ユーザーの記事の場合、「編集」「削除」ボタンが表示されないこと

### 3-4. 記事編集

1. 自分が作成した記事の詳細ページを開く
2. 「編集」ボタンをクリック
3. タイトルを変更して「更新する」ボタンをクリック
4. **確認事項**:
   - 記事詳細ページにリダイレクトされること
   - 変更後のタイトルが反映されていること

### 3-5. 記事削除

1. 自分が作成した記事の詳細ページを開く
2. 「削除」ボタンをクリック
3. **確認事項**:
   - 記事一覧ページにリダイレクトされること
   - 削除した記事が一覧に表示されないこと

### 3-6. 他ユーザーの記事の編集・削除制限

1. 別のメールアドレスで新規ユーザー登録
2. 最初のユーザーが作成した記事の詳細ページにアクセス
3. **確認事項**:
   - 「編集」「削除」ボタンが表示されないこと
   - 直接 `PUT /posts/:id` や `DELETE /posts/:id` にリクエストしても 403 が返ること

---

## テストシナリオ 4: コメント機能

前提: ログイン済み状態、記事が 1 件以上存在

### 4-1. コメント投稿

1. 任意の記事詳細ページを開く
2. コメントフォームに内容を入力して「コメントを投稿」ボタンをクリック
3. **確認事項**:
   - 記事詳細ページがリロードされること（または SPA 更新）
   - 投稿したコメントが表示されること
   - コメント者名・投稿日時が表示されること

### 4-2. 空コメントのバリデーション

1. 記事詳細ページでコメントフォームを空のまま送信
2. **確認事項**:
   - バリデーションエラーが表示されること

---

## テストシナリオ 5: Inertia.js SPA ナビゲーション

### 5-1. ページ遷移のパフォーマンス確認

1. 開発者ツールの「ネットワーク」タブを開く
2. 記事一覧から記事詳細ページへ遷移
3. **確認事項**:
   - フルページリロードではなく XHR/Fetch リクエストで遷移していること
   - `X-Inertia: true` ヘッダーがリクエストに含まれること
   - レスポンスが JSON 形式（Inertia プロトコル）であること

### 5-2. ブラウザバック

1. 記事詳細ページから「戻る」ボタン（ブラウザ）を押す
2. **確認事項**:
   - 正しく前のページ（記事一覧）に戻ること

---

## テストシナリオ 6: セキュリティ確認

### 6-1. セッション Cookie の設定確認

1. ログイン後、開発者ツール > Application > Cookies を確認
2. **確認事項**:
   - Cookie 名: `session`
   - `HttpOnly` が設定されていること（JS からアクセス不可）
   - `SameSite=Lax` または `Strict` が設定されていること

### 6-2. XSS 対策の確認

1. コメントフォームに以下を入力して送信:
   ```
   <script>alert('XSS')</script>
   ```
2. **確認事項**:
   - アラートが表示されないこと
   - テキストとしてエスケープされて表示されること

---

## テストシナリオ 7: レスポンシブデザイン

1. Chrome DevTools でモバイルエミュレーター（iPhone SE など）に切り替え
2. 各ページを確認
3. **確認事項**:
   - ナビゲーションメニューが適切に折りたたまれること
   - コンテンツが横スクロールなしで表示されること
   - フォームが使いやすいサイズで表示されること
